## 1. 需求概述

### 1.1 核心需求
- ✅ **q/a 字段保持原文不变**，用户数据100%安全
- ✅ 对 `dataset_datas` 的所有 `indexes` 内容进行同义词替换并存储
- ✅ 全文检索和向量检索使用同义词替换后的内容
- ✅ 记录同义词转换关系（synonymMetadata），支持原文恢复
- ✅ 当用户上传或删除同义词时，异步触发数据标准化或恢复
- ⚠️ **用户自定义数据（手动添加）默认不做标准化**

### 1.2 需要覆盖的场景

| 场景 | 描述 | 处理方式 |
|-----|------|---------|
| **场景1** | 知识库无同义词 → 首次上传同义词 | 标准化所有**系统生成**的数据 |
| **场景2** | 知识库有同义词A → 更新为同义词B | 先恢复A，再用B标准化 |
| **场景3** | 知识库有同义词 → 删除同义词 | 恢复所有数据为原文 |
| **场景4** | 知识库有同义词 → 用户新增自定义分片 | 根据开关决定是否标准化（**默认不标准化**） |
| **场景5** | 知识库有同义词 → 用户新增自定义增强索引 | 根据开关决定是否标准化（**默认不标准化**） |
| **场景6** | 知识库有同义词 → 系统生成新数据（上传文件等） | **自动标准化** |
| **场景7** | 直连数据库类型的知识库 | **不做标准化**，维持原样 |

### 1.3 数据来源分类

```
数据来源
├── 系统生成（默认标准化）
│   ├── 文件上传解析
│   ├── QA拆分生成
│   ├── 增强索引（hype、summary、question、small2Big）
│   └── 图片描述索引
│
├── 用户自定义（默认不标准化，开关控制）
│   ├── 手动新增分片（virtual collection）
│   ├── 手动编辑 q/a
│   └── 手动新增自定义索引（type: custom）
│
└── 直连数据库（不做标准化）
    └── 外部数据库同步的数据（MySQL、PostgreSQL 等直连类型）
```

### 1.4 数据结构变更
1. **dataset_datas 的 q/a 字段**：**保持原文不变**，前端展示原文
2. **dataset_datas 外层**：新增 `skipSynonymStandardize` 字段，标记整个分片是否跳过同义词标准化
3. **indexes 内每个元素**：修改 `text` 字段存储标准化后的文本
4. **indexes 内每个元素**：新增 `synonymMetadata` 字段，记录该 index 的同义词转换关系
5. **indexes 内每个元素**：新增 `skipSynonymStandardize` 字段，标记索引级别是否跳过同义词标准化
6. **dataset_data_texts 表**：`fullTextToken` 使用标准化后的 q/a 生成

## 3. 方案设计

### 3.1 能否复用 dataset_trainings？

**结论：不建议复用，需要新建独立队列表**

#### 3.1.1 对比分析

| 对比维度 | dataset_trainings | 同义词处理队列 |
|---------|-------------------|---------------|
| **处理对象** | 原始文本 → 新数据 | 已有数据 → 更新数据 |
| **处理流程** | 分词、QA生成、向量化 | 同义词替换/恢复、向量更新 |
| **数据关系** | 创建新记录 | 更新现有记录 |
| **回滚需求** | 无 | 需要支持恢复原文 |
| **mode类型** | chunk/qa/image等7种 | standardize/restore 2种 |
| **依赖数据** | 原始文本 | dataId + synonymFileIds |
| **失败重试** | 用户可见，有重试入口 | ⚠️ **这是关键问题** |

#### 3.1.5 重试API设计

```typescript
// POST /api/core/dataset/synonym/queue/retry
export async function retrySynonymQueue({
  queueId
}: {
  queueId: string;
}) {
  // 1. 找到失败的队列任务
  const queue = await MongoDatasetSynonymQueue.findById(queueId);

  if (queue.status !== 'failed') {
    throw new Error('只能重试失败的任务');
  }

  // 2. 重置状态，从断点继续
  await MongoDatasetSynonymQueue.updateOne(
    { _id: queueId },
    {
      $set: {
        status: 'pending',
        lockTime: new Date('2000/1/1'),
        retryCount: 3  // 重置重试次数
      }
    }
  );

  // 3. Worker 会自动捡起这个任务
  // 从 lastProcessedDataId 继续处理
}
```

#### 3.2.1 修改 dataset_datas 结构

```typescript
// packages/global/core/dataset/type.d.ts

/**
 * 单次同义词转换记录
 * 用于记录原文到标准词的转换，支持精确恢复
 *
 * 注意：与现有 packages/service/core/dataset/indexTransform/utils.ts 中的定义保持一致

export type TransformationRecordType = {
  // 在原始文本中的位置信息
  originalStartPos: number;    // 原始文本中的起始位置（字符索引）
  originalEndPos: number;      // 原始文本中的结束位置（字符索引）
  originalTerm: string;        // 原始词汇（非标准词/同义词）

  // 在转换后文本中的位置信息
  transformedStartPos: number; // 转换后文本中的起始位置（字符索引）
  transformedEndPos: number;   // 转换后文本中的结束位置（字符索引）
  standardizedTerm: string;    // 标准化词

  // 关联信息
  synonymMappingId: string;    // 关联的同义词映射记录ID（MongoSynonymMapping._id）
};

/**
 * 同义词转换元数据
 * 存储在 indexes 的 synonymMetadata 字段中
 */
export type SynonymMetadataType = {
  synonymFileIds: string[];                     // 关联的同义词文件ID数组（支持多个同义词文件）
  transformations: TransformationRecordType[];  // 转换记录数组
};

export type DatasetDataIndexItemType = {
  type: `${DatasetDataIndexTypeEnum}`;
  dataId: string;
  text: string;  // ⚠️ 存储标准化后的文本（应用了同义词）

  // 新增：该index的同义词转换元数据
  synonymMetadata?: SynonymMetadataType;

  // 新增：跳过同义词标准化标记（针对单个索引）
  // true = 用户自定义索引，不参与同义词标准化
  // false/undefined = 参与同义词标准化
  skipSynonymStandardize?: boolean;
};

export type DatasetDataSchemaType = DatasetDataFieldType & {
  // ✅ q/a 保持原文不变
  q: string;  // 用户看到的原文
  a: string;  // 用户看到的原文

  // 新增：跳过同义词标准化标记（分片级别）
  // true = 用户自定义数据，不参与同义词标准化
  // false/undefined = 系统生成数据，参与同义词标准化
  skipSynonymStandardize?: boolean;

  indexes: DatasetDataIndexItemType[];
};
```

```typescript
  column_val_index = '...',      // 数据库列值 → 系统生成，需标准化
  hype = 'hype',                 // 增强索引 → 系统生成，需标准化
  small2Big = 'small2Big'        // 小转大索引 → 系统生成，需标准化
}
```

**标准化策略**：
- `custom` 类型的 index：根据 `skipSynonymStandardize` 开关决定
- 其他类型的 index：**始终标准化**

**LLM生成类索引的处理流程**（hype、summary、question等）：

这类索引是**后验生成**的，即数据先入库，后台任务再读取数据生成索引。

1. **数据入库阶段**：q/a 先经过实时标准化后入库
2. **后验生成阶段**：Hype等任务读取**已标准化的q/a**调用LLM生成索引
3. **索引存储阶段**：生成的索引直接存入（已基于标准化内容，无需再标准化）

```
T1: 原始数据 q="人工智能应用"
    ↓ 实时标准化
T2: 入库 q="AI应用"（标准化后）
    ↓ Hype后验任务
T3: 读取 q="AI应用"，调用LLM生成逆向问题
    ↓
T4: LLM生成: "什么是AI应用？"（已基于标准化内容）
    ↓
T5: 直接存入 indexes（无需再标准化）
```

**优势**：
- Hype生成的内容与主数据语义一致
- 生成的indexes无需再做同义词替换，逻辑更简单
- 用户搜索"AI应用"时，主数据和Hype索引都能匹配

### 3.3 处理流程设计

#### 3.3.1 同义词上传流程

```
用户上传同义词文件
       ↓

       ↓
创建 synonym_queue 任务（mode: standardize）
       ↓
后台Worker异步处理：
  ├─ 1. 获取队列任务
  ├─ 2. 分批查询 dataset_datas（按 lastProcessedDataId 分页）
  ├─ 3. 对每条数据：
  │    ├─ 应用同义词转换到 q、a
  │    ├─ 应用同义词转换到每个 index.text
  │    ├─ 记录 synonymMetadata（保存原文恢复信息）
  │    ├─ 更新向量存储（PG/Milvus）
  │    └─ 更新全文检索索引
  ├─ 4. 更新进度（progress.processed）
  ├─ 5. 处理完成后标记 status: completed
  └─ 6. 失败时记录错误，支持重试
```

#### 3.3.2 同义词更新流程（场景2）

```typescript
// packages/service/worker/synonym/synonymQueueWorker.ts

import { MongoDatasetSynonymQueue } from '../../core/dataset/synonymQueue/schema';
import { processSynonymQueueTask } from './processor';
import { addLog } from '../../common/system/log';

export const runSynonymQueueWorker = async () => {
  const LOCK_MINUTES = 5;
  const BATCH_SIZE = 100;

  // 1. 获取待处理任务（锁定）
  const task = await MongoDatasetSynonymQueue.findOneAndUpdate(
    {
      status: { $in: ['pending', 'processing'] },
      lockTime: { $lt: new Date(Date.now() - LOCK_MINUTES * 60 * 1000) },
      retryCount: { $gt: 0 }
    },
    {
      $set: {
        status: 'processing',
        lockTime: new Date(),
        startTime: new Date()
      }
    },
    { sort: { createTime: 1 }, new: true }
  );

  if (!task) return;

  try {
    // 2. 处理任务
    await processSynonymQueueTask(task, BATCH_SIZE);

    // 3. 标记完成
    await MongoDatasetSynonymQueue.updateOne(
      { _id: task._id },
      {
        $set: {
          status: 'completed',
          endTime: new Date()
        }
      }
    );
  } catch (error) {
    addLog.error(`[SynonymQueue] Task ${task._id} failed:`, error);

    // 4. 错误处理，减少重试次数
    await MongoDatasetSynonymQueue.updateOne(
      { _id: task._id },
      {
        $set: {
          status: task.retryCount <= 1 ? 'failed' : 'pending',
          lockTime: new Date('2000/1/1'),
          errorMsg: String(error)
        },
        $inc: { retryCount: -1 }
      }
    );
  }
};
```

#### 3.4.2 任务处理器

```typescript
// packages/service/worker/synonym/processor.ts

import { MongoDatasetData } from '../../core/dataset/data/schema';
import { MongoDataset } from '../../core/dataset/schema';
import { MongoDatasetSynonymQueue } from '../../core/dataset/synonymQueue/schema';
import { getDatasetSynonymConfig } from '../../core/dataset/indexTransform/controller';
import { applySynonymTransform, restoreOriginalText } from '../../core/dataset/indexTransform/utils';
import { batchUpdateDataVector, updateDataFullText } from './vectorUpdater';

export const processSynonymQueueTask = async (
  task: DatasetSynonymQueueType,
  batchSize: number
) => {
  const { teamId, datasetId, synonymFileIds, oldSynonymFileIds, mode, lastProcessedDataId } = task;

  // ⚠️ 获取知识库配置（需要 vectorModel 用于向量更新）
  const dataset = await MongoDataset.findById(datasetId)
    .select('vectorModel')
    .lean();
  if (!dataset) {
    throw new Error(`Dataset ${datasetId} not found`);
  }

  // 获取同义词配置
  const synonymConfig = (mode === 'standardize' || mode === 'update')
    ? await getDatasetSynonymConfig({ teamId: String(teamId), datasetId: String(datasetId) })
    : null;

  let processedCount = task.progress.processed;
  let cursor = lastProcessedDataId;

  // update 模式分两阶段：先恢复，再标准化
  if (mode === 'update') {
    // Phase 1: 恢复旧同义词的数据（遍历所有旧文件ID）
    await restoreByFileIds(teamId, datasetId, oldSynonymFileIds!, batchSize, dataset);
    // Phase 2: 重置游标，从头开始标准化
    cursor = undefined;
  }

  while (true) {
    // 分批查询数据
    const query: any = {
      teamId,
      datasetId,
      ...(mode === 'restore'
        ? {
            // 恢复模式：只查询有 synonymMetadata 且 synonymFileIds 包含目标文件ID的数据
            $or: [
              { 'synonymMetadata.q.synonymFileIds': { $in: synonymFileIds } },
              { 'synonymMetadata.a.synonymFileIds': { $in: synonymFileIds } },
              { 'indexes.synonymMetadata.synonymFileIds': { $in: synonymFileIds } }
            ]
          }
        : {
            // 标准化模式：排除 skipSynonymStandardize=true 的数据
            skipSynonymStandardize: { $ne: true }
          }),
      ...(cursor ? { _id: { $gt: cursor } } : {})
    };

    const dataList = await MongoDatasetData.find(query)
      .sort({ _id: 1 })
      .limit(batchSize)
      .lean();

    if (dataList.length === 0) break;

    // 批量处理（传入 dataset 用于获取 vectorModel）
    for (const data of dataList) {
      if (mode === 'standardize' || mode === 'update') {
        await standardizeDataItem(data, synonymConfig!, synonymFileIds.map(String), dataset);
      } else {
        await restoreDataItem(data, dataset);
      }
      processedCount++;
    }

    // 更新进度
    cursor = dataList[dataList.length - 1]._id;
    await MongoDatasetSynonymQueue.updateOne(
      { _id: task._id },
      {
        $set: {
          'progress.processed': processedCount,
          lastProcessedDataId: cursor,
          lockTime: new Date()  // 续期锁
        }
      }
    );
  }
};

// 按文件ID数组恢复数据（用于 update 模式的第一阶段）
const restoreByFileIds = async (
  teamId: ObjectId,
  datasetId: ObjectId,
  synonymFileIds: ObjectId[],
  batchSize: number,
  dataset: { vectorModel: string }  // ⚠️ 添加 dataset 参数，用于向量更新
) => {
  let cursor: ObjectId | undefined;

  while (true) {
    const dataList = await MongoDatasetData.find({
      teamId,
      datasetId,
      $or: [
        { 'synonymMetadata.q.synonymFileIds': { $in: synonymFileIds } },
        { 'synonymMetadata.a.synonymFileIds': { $in: synonymFileIds } },
        { 'indexes.synonymMetadata.synonymFileIds': { $in: synonymFileIds } }
      ],
      ...(cursor ? { _id: { $gt: cursor } } : {})
    })
      .sort({ _id: 1 })
      .limit(batchSize)
      .lean();

    if (dataList.length === 0) break;

    for (const data of dataList) {
      await restoreDataItem(data, dataset);
    }

    cursor = dataList[dataList.length - 1]._id;
  }
};

/**
 * 标准化单条数据
 * ⚠️ 核心变更：
 * 1. q/a 字段保持原文不变（不修改存储）
 * 2. 只对 indexes[].text 应用同义词并存储
 * 3. 全文检索使用标准化后的 q/a 生成 token
 * 4. 向量更新采用"先删后插"策略，会产生新的 dataId
 */
const standardizeDataItem = async (
  data: DatasetDataSchemaType,
  synonymConfig: SynonymConfigType,
  synonymFileIds: string[],
  dataset: { vectorModel: string },
  task: { tmbId: ObjectId; billId?: ObjectId }  // ← 新增：用于费用计算
) => {
  const { synonymDict, synonymMappingMap } = synonymConfig;

  // 1. ✅ 对 q/a 应用同义词转换（仅用于全文检索，不存储到数据库）
  const qStandardized = applySynonymTransform(data.q, synonymDict, synonymMappingMap);
  const aStandardized = data.a
    ? applySynonymTransform(data.a, synonymDict, synonymMappingMap)
    : null;

  // 2. ✅ 处理每个 index（收集需要更新向量的项）
  const indexesToUpdate: Array<{ index: any; indexResult: any; arrayIndex: number }> = [];

  for (let i = 0; i < data.indexes.length; i++) {
    const index = data.indexes[i];

    // ⚠️ 跳过标记为不标准化的索引（用户自定义数据）
    if (index.skipSynonymStandardize === true) continue;

    const indexResult = applySynonymTransform(index.text, synonymDict, synonymMappingMap);
    if (indexResult.transformations.length > 0) {
      indexesToUpdate.push({ index, indexResult, arrayIndex: i });
    }
  }

  // 3. ✅ 批量更新向量（先删后插，会产生新的 dataId）
  let newDataIds: string[] = [];
  let tokens = 0;

  if (indexesToUpdate.length > 0) {
    const result = await batchUpdateDataVector({
      teamId: String(data.teamId),
      datasetId: String(data.datasetId),
      collectionId: String(data.collectionId),
      items: indexesToUpdate.map(item => ({
        oldDataId: item.index.dataId,
        newText: item.indexResult.transformedText  // ← 标准化后的文本
      })),
      embeddingModel: dataset.vectorModel
    });

    newDataIds = result.insertIds;
    tokens = result.tokens;
  }

  // 4. ✅ 构建新的 indexes 数组（使用新的 dataId）
  const newIndexes = data.indexes.map((index, i) => {
    const updateIndex = indexesToUpdate.findIndex(item => item.arrayIndex === i);

    if (updateIndex !== -1) {
      const { indexResult } = indexesToUpdate[updateIndex];
      return {
        ...index,
        dataId: newDataIds[updateIndex],  // ← 使用新的 dataId
        text: indexResult.transformedText,  // ← 存储标准化后的文本
        synonymMetadata: {
          synonymFileIds,
          transformations: indexResult.transformations
        }
      };
    }

    return index;  // 未变化的保持原样
  });

  // 5. ✅ 更新数据库（q/a 保持原文不变）
  await MongoDatasetData.updateOne(
    { _id: data._id },
    {
      $set: {
        // ⚠️ 核心变更：q/a 不修改，保持原文
        // q: data.q,  ← 不更新
        // a: data.a,  ← 不更新
        indexes: newIndexes
      }
    }
  );

  // 6. ✅ 更新全文检索（使用标准化后的 q/a）
  await updateDataFullText({
    dataId: String(data._id),
    datasetId: String(data.datasetId),
    collectionId: String(data.collectionId),
    teamId: String(data.teamId),
    standardizedQ: qStandardized.transformedText,  // ← 使用标准化后的文本
    standardizedA: aStandardized?.transformedText
  });

  // 7. ✅ 推送费用记录
  if (tokens > 0) {
    pushGenerateVectorUsage({
      teamId: String(data.teamId),
      tmbId: String(task.tmbId),
      inputTokens: tokens,
      model: dataset.vectorModel,
      billId: task.billId
    });
  }
};

/**
 * 恢复单条数据
 * ⚠️ 核心变更：
 * 1. q/a 字段本来就是原文，不需要恢复
 * 2. 只恢复 indexes[].text 为原文
 * 3. 全文检索恢复为原文 token
 * 4. 向量更新采用"先删后插"策略，会产生新的 dataId
 */
const restoreDataItem = async (
  data: DatasetDataSchemaType,
  dataset: { vectorModel: string }
) => {
  // 1. ✅ q/a 本来就是原文，不需要恢复

  // 2. ✅ 收集需要恢复向量的 index
  const indexesToRestore: Array<{ index: any; originalText: string; arrayIndex: number }> = [];

  for (let i = 0; i < data.indexes.length; i++) {
    const index = data.indexes[i];
    if (index.synonymMetadata) {
      const originalText = restoreOriginalText(index.text, index.synonymMetadata.transformations);
      indexesToRestore.push({ index, originalText, arrayIndex: i });
    }
  }

  // 3. ✅ 批量更新向量
  let newDataIds: string[] = [];
  if (indexesToRestore.length > 0) {
    const result = await batchUpdateDataVector({
      teamId: String(data.teamId),
      datasetId: String(data.datasetId),
      collectionId: String(data.collectionId),
      items: indexesToRestore.map(item => ({
        oldDataId: item.index.dataId,
        newText: item.originalText  // ← 恢复的原文
      })),
      embeddingModel: dataset.vectorModel
    });

    newDataIds = result.insertIds;
  }

  // 4. ✅ 构建新的 indexes 数组
  const newIndexes = data.indexes.map((index, i) => {
    const restoreIndex = indexesToRestore.findIndex(item => item.arrayIndex === i);

    if (restoreIndex !== -1) {
      return {
        type: index.type,
        dataId: newDataIds[restoreIndex],  // ← 使用新的 dataId
        text: indexesToRestore[restoreIndex].originalText  // ← 恢复的原文
        // 移除 synonymMetadata
      };
    }

    // 未变化的保持原样（但移除 synonymMetadata）
    const { synonymMetadata, ...rest } = index;
    return rest;
  });

  // 5. ✅ 更新数据库（q/a 不需要修改）
  await MongoDatasetData.updateOne(
    { _id: data._id },
    {
      $set: {
        // ⚠️ 核心变更：q/a 本来就是原文，不需要修改
        // q: data.q,  ← 不更新
        // a: data.a,  ← 不更新
        indexes: newIndexes
      }
    }
  );

  // 6. ✅ 更新全文检索（使用原文 q/a）
  await updateDataFullText({
    dataId: String(data._id),
    datasetId: String(data.datasetId),
    collectionId: String(data.collectionId),
    teamId: String(data.teamId),
    standardizedQ: data.q,  // ← 使用原文
    standardizedA: data.a
  });
};
```

## 4. 新增数据实时标准化

### 4.1 实时标准化工具函数

```typescript
// packages/service/core/dataset/synonym/realtime.ts

import { getDatasetSynonymConfig } from '../indexTransform/controller';
import { applySynonymTransform } from '../indexTransform/utils';
import type { DatasetDataIndexItemType } from '@fastgpt/global/core/dataset/type';

/**
 * 实时标准化数据（用于新增数据时）
 *
 * @param params.teamId - 团队ID
 * @param params.datasetId - 知识库ID
 * @param params.q - 问题/内容
 * @param params.a - 答案（可选）
 * @param params.indexes - 索引数组
 * @param params.isUserCustom - 是否为用户自定义数据
 * @param params.applySynonym - 用户是否选择应用同义词（仅当 isUserCustom=true 时生效）
 *
 * @returns 标准化后的数据，包含 synonymMetadata
 */
export async function realtimeSynonymStandardize({
  teamId,
  datasetId,
  q,
  a,
  indexes,
  isUserCustom = false,
  applySynonym = false
}: {
  teamId: string;
  datasetId: string;
  q: string;
  a?: string;
  indexes: Omit<DatasetDataIndexItemType, 'dataId'>[];
  isUserCustom?: boolean;
  applySynonym?: boolean;
}): Promise<{
  q: string;
  a?: string;
  indexes: Omit<DatasetDataIndexItemType, 'dataId'>[];
  synonymMetadata?: any;
  skipSynonymStandardize?: boolean;
}> {
  // 1. 获取知识库的同义词配置
  const synonymConfig = await getDatasetSynonymConfig({ teamId, datasetId });

  // 2. 如果没有同义词配置，直接返回原数据
  if (!synonymConfig) {
    return { q, a, indexes };
  }

  const { synonymDict, synonymMappingMap, synonymFileIds } = synonymConfig;  // ⚠️ 改为数组

  // 3. 用户自定义数据 + 不应用同义词 → 标记跳过
  if (isUserCustom && !applySynonym) {
    return {
      q,
      a,
      indexes,
      skipSynonymStandardize: true
    };
  }

  // 4. 执行同义词标准化
  const qResult = applySynonymTransform(q, synonymDict, synonymMappingMap);
  const aResult = a ? applySynonymTransform(a, synonymDict, synonymMappingMap) : null;

  const newIndexes = indexes.map((index) => {
    // custom 类型 + 用户自定义 + 不应用 → 跳过
    if (index.type === 'custom' && isUserCustom && !applySynonym) {
      return index;
    }

    const indexResult = applySynonymTransform(index.text, synonymDict, synonymMappingMap);
    return {
      ...index,
      text: indexResult.transformedText,
      ...(indexResult.transformations.length > 0 && {
        synonymMetadata: {
          synonymFileIds,  // ⚠️ 使用数组
          transformations: indexResult.transformations
        }
      })
    };
  });

  return {
    q: qResult.transformedText,
    a: aResult?.transformedText ?? a,
    indexes: newIndexes,
    synonymMetadata: {
      ...(qResult.transformations.length > 0 && {
        q: { synonymFileIds, transformations: qResult.transformations }  // ⚠️ 使用数组
      }),
      ...(aResult?.transformations.length && {
        a: { synonymFileIds, transformations: aResult.transformations }  // ⚠️ 使用数组
      })
    },
    skipSynonymStandardize: false
  };
}
```

### 5.1 向量更新

> ⚠️ **重要**：向量数据库（PG/Milvus）没有直接更新向量的能力，需要采用"先删后插"策略。
> 由于删除后重新插入会产生**新的 dataId**，需要在调用方更新 MongoDB 中的引用。

```typescript
// packages/service/worker/synonym/vectorUpdater.ts

import { deleteDatasetDataVector, insertDatasetDataVector } from '../../../common/vectorDB/controller';
import { getVectorsByText } from '../../ai/embedding';
import { getEmbeddingModel } from '../../ai/model';

/**
 * 更新向量数据（先删后插策略）
 * @returns 新的 dataId（因为是重新插入，ID会变化）
 */
export const updateDataVector = async ({
  teamId,
  datasetId,
  collectionId,
  oldDataId,
  newText,
  embeddingModel
}: {
  teamId: string;
  datasetId: string;
  collectionId: string;
  oldDataId: string;
  newText: string;
  embeddingModel: string;
}): Promise<string> => {
  // 1. 删除旧向量
  await deleteDatasetDataVector({
    teamId,
    idList: [oldDataId]
  });

  // 2. 生成新向量并插入
  const { insertIds } = await insertDatasetDataVector({
    teamId,
    datasetId,
    collectionId,
    inputs: [newText],
    model: getEmbeddingModel(embeddingModel)
  });

  // 3. 返回新的 dataId
  return insertIds[0];
};

/**
 * 批量更新向量数据（优化版本，减少API调用）
 * @returns 新的 dataId 数组
 */
export const batchUpdateDataVector = async ({
  teamId,
  datasetId,
  collectionId,
  items,
  embeddingModel
}: {
  teamId: string;
  datasetId: string;
  collectionId: string;
  items: Array<{ oldDataId: string; newText: string }>;
  embeddingModel: string;
}): Promise<string[]> => {
  if (items.length === 0) return [];

  // 1. 批量删除旧向量
  await deleteDatasetDataVector({
    teamId,
    idList: items.map(item => item.oldDataId)
  });

  // 2. 批量生成新向量并插入
  const { insertIds } = await insertDatasetDataVector({
    teamId,
    datasetId,
    collectionId,
    inputs: items.map(item => item.newText),
    model: getEmbeddingModel(embeddingModel)
  });

  return insertIds;
};
```

## 9. 风险与注意事项

### 9.1 与 dataset_trainings 队列的并发处理 ⚠️

#### 9.1.1 冲突场景分析

| 场景 | 描述 | 风险 | 解决方案 |
|-----|------|-----|---------|
| **场景A** | training队列创建新数据，同义词队列正在扫描 | 新数据可能被遗漏 | 新数据通过实时标准化处理，不依赖队列 |
| **场景B** | 同义词队列处理存量数据，Hype同时生成增强索引 | Hype基于未标准化内容生成 | **知识库级别锁**，同义词处理期间暂停Hype |
| **场景C** | 两个队列同时更新同一条数据 | 数据不一致 | **知识库级别锁** |
| **场景D** | 同义词恢复时，training正在用标准化内容生成 | 生成内容基于错误的源 | **知识库级别锁** |
| **场景E** | training队列正在处理chunk，用户上传同义词文件 | 部分数据标准化、部分未标准化 | **补漏扫描** + 实时标准化配合 |

**场景详细说明与用户操作示例：**

**场景E：training队列正在处理chunk，用户同时上传同义词文件**

- **用户操作**：用户正在上传一个大文件（如 PDF），training 队列正在分批处理 chunk，此时用户又上传了同义词文件
- **时间线示例（问题场景）**：
  ```
  T0: 用户上传大文件，training队列开始处理，预计生成1000条数据
  T1: training队列已处理完300条数据（已入库）
  T2: 用户上传同义词文件，创建同义词队列任务
  T3: 同义词队列开始扫描，发现300条存量数据，开始标准化
  T4: training队列继续处理，又生成了200条数据（第301-500条）
      - 此时知识库已有同义词配置
      - 问题：这200条数据是否会被标准化？
  T5: 同义词队列完成300条存量数据的标准化
  T6: training队列继续处理剩余500条数据（第501-1000条）
  ```
- **解决方案**：**实时标准化 + 补漏扫描** 双重保障
  1. **实时标准化**：T4 时刻，training 队列创建数据时检测到知识库已有同义词配置，自动进行实时标准化
  2. **补漏扫描**：同义词队列完成主处理后，扫描 `createTime >= task.startTime` 且未标准化的数据

- **正确的时间线**：
  ```
  T0: 用户上传大文件，training队列开始处理
  T1: training队列处理完300条数据，正常入库（无同义词配置）
  T2: 用户上传同义词文件
      - 保存同义词配置到 dataset_synonyms
      - 创建同义词队列任务
      - 设置 dataset.synonymProcessingStatus = 'processing'
  T3: 同义词队列开始扫描存量数据（300条）
  T4: training队列继续处理，创建新数据时：
      - 检测到 synonymProcessingStatus = 'processing'
      - 策略2：跳过实时标准化，标记 pendingSynonymProcess = true
      - 数据正常入库，等待补漏扫描处理
  T5: 同义词队列完成存量数据标准化
  T6: 同义词队列执行补漏扫描
      - 查询 createTime >= startTime 且 synonymMetadata 为空的数据
      - 对 T4 期间创建的数据进行标准化
  T7: training队列继续处理剩余数据
      - 此时 synonymProcessingStatus 已变为 'idle'
      - 新数据直接进行实时标准化
  结果: 所有数据都被正确标准化 ✓
  ```

**关键点**：
1. 同义词队列处理期间，training 创建的新数据**不做实时标准化**（避免配置不一致）
2. 同义词队列完成后的**补漏扫描**会处理这些遗漏的数据
3. 补漏扫描完成后，`synonymProcessingStatus` 才变为 `idle`

---

**场景A：training队列创建新数据，同义词队列正在扫描**

- **用户操作**：用户上传同义词文件后，在同义词队列还在处理的过程中，又上传了一个新的PDF文件到同一个知识库
- **时间线示例（无实时标准化时的问题）**：
  ```
  T0: 用户上传同义词文件，创建同义词队列任务
  T1: 同义词队列开始扫描，发现1000条数据需要处理
  T2: 同义词队列正在处理第500条数据
  T3: 用户上传新的PDF文件
  T4: training队列开始处理PDF，创建了200条新的dataset_data
  T5: 同义词队列扫描完成（只处理了原来的1000条）
  结果: 新上传的200条数据未被同义词标准化！
  ```
- **解决方案**：新数据通过实时标准化处理，不依赖队列
- **正确的时间线（有实时标准化）**：
  ```
  T0: 用户上传同义词文件，创建同义词队列任务（处理存量数据）
  T1: 同义词队列开始扫描存量数据
  T3: 用户上传新的PDF文件
  T4: training队列处理PDF，创建dataset_data时：
      - 检测到知识库已启用同义词标准化
      - 在insertDatasetData时直接调用实时标准化逻辑
      - 新数据入库时已经是标准化后的内容
  T5: 同义词队列继续处理存量数据
  结果: 新数据在入库时已完成标准化，无需队列处理 ✓
  ```

**场景B：同义词队列处理存量数据，Hype同时为该数据生成增强索引**

- **用户操作**：用户之前已启用Hype增强索引功能，现在又上传了同义词文件，两个队列同时在后台运行
- **时间线示例（问题场景）**：
  ```
  T0: 知识库有100条存量数据（q="人工智能应用"），已启用Hype增强索引
  T1: 用户上传同义词文件，同义词队列开始处理存量数据
  T2: Hype任务读取第50条数据，此时q还是"人工智能应用"（同义词队列还没处理到）
       - LLM基于"人工智能应用"生成逆向问题: "什么是人工智能应用？"
       - 保存到indexes
  T3: 同义词队列将第50条数据的q标准化为"AI应用"
  结果: 主数据q="AI应用"，但Hype索引="什么是人工智能应用？"（基于旧内容生成）
  ```
- **风险等级**：中等（功能不一致，但不会导致数据丢失）
- **问题本质**：这是**并发竞争**问题，Hype读取数据时，该数据还未被同义词队列处理
- **解决方案**：使用知识库级别锁，同义词队列处理期间暂停该知识库的Hype任务
- **正确的时间线（有知识库锁）**：
  ```
  T0: 知识库有100条存量数据（q="人工智能应用"）
  T1: 用户上传同义词文件，同义词队列获取知识库锁，开始处理
  T2: Hype任务检测到知识库锁，暂停等待
  T3: 同义词队列处理完成，释放锁，所有数据q已标准化为"AI应用"
  T4: Hype任务获取锁，读取q="AI应用"（已标准化）
       - LLM基于"AI应用"生成逆向问题: "什么是AI应用？"
       - 保存到indexes
  结果: 主数据q="AI应用"，Hype索引="什么是AI应用？" ✓
  ```

**场景C：两个队列同时更新同一条数据**

- **用户操作**：用户在同义词处理过程中，手动编辑了某条正在被处理的数据（编辑数据会创建training任务重新生成向量）
- **时间线示例**：
  ```
  T0: 同义词队列正在处理第100条数据
       - 读取数据: q="机器学习算法"
       - 正在进行同义词替换...
  T1: 用户在前端编辑第100条数据的内容
       - 用户将q改为"深度学习模型"
       - 前端调用更新API，创建training任务重新生成向量
       - training任务写入数据库: q="深度学习模型"
  T2: 同义词队列处理完成
       - 基于旧的q="机器学习算法"生成标准化结果"ML算法"
       - 写入数据库: q="ML算法"，覆盖了用户的修改
  结果: 用户的编辑被覆盖了！q变成了"ML算法"而不是用户修改的"深度学习模型"
  ```
- **风险等级**：高（用户数据丢失）
- **解决方案**：知识库级别的处理状态锁，同义词处理期间暂停该知识库的training任务
- **补充方案**：同义词队列更新数据时使用乐观锁，检查数据版本号或updateTime，若已被修改则跳过该条数据

**场景D：同义词恢复时，training正在用标准化内容生成**

- **用户操作**：用户删除同义词库触发恢复流程，同时training队列正在为某条数据生成索引
- **时间线示例**：
  ```
  T0: 知识库数据已被同义词标准化
       - 第50条数据: q="AI应用", synonymMetadata记录原始为"人工智能应用"
  T1: 用户删除同义词库，触发恢复队列任务
  T2: 恢复队列正在处理第50条数据
       - 读取synonymMetadata，准备恢复...
  T3: 同时，training队列为第50条数据生成Hype增强索引
       - 读取当前q="AI应用"（标准化后的内容）
       - 基于"AI应用"生成增强内容并保存
  T4: 恢复队列完成
       - 将q恢复为"人工智能应用"
  结果: Hype增强索引是基于"AI应用"生成的，但主数据已恢复为"人工智能应用"，语义不一致！
  ```
- **风险等级**：高（数据语义不一致）
- **解决方案**：恢复期间同样需要知识库级别锁，确保恢复完成后再进行其他索引生成

#### 9.1.2 解决方案：知识库级别的处理状态锁

**核心思路**：在知识库（Dataset）级别增加处理状态标记，两个队列互相感知。

**方案一：在 Dataset 表增加状态字段（推荐）**

**处理逻辑**：

```typescript
// 同义词队列开始处理前
const startSynonymProcessing = async (datasetId: string, taskId: string) => {
  const result = await MongoDataset.findOneAndUpdate(
    {
      _id: datasetId,
      $or: [
        { 'synonymProcessingStatus.status': { $ne: 'processing' } },
        { 'synonymProcessingStatus.status': { $exists: false } }
      ]
    },
    {
      $set: {
        'synonymProcessingStatus.status': 'processing',
        'synonymProcessingStatus.taskId': taskId,
        'synonymProcessingStatus.startTime': new Date()
      }
    },
    { new: true }
  );

  if (!result) {
    throw new Error('Dataset is being processed by another task');
  }
};

// 同义词队列处理完成后
const endSynonymProcessing = async (datasetId: string) => {
  await MongoDataset.updateOne(
    { _id: datasetId },
    {
      $set: { 'synonymProcessingStatus.status': 'idle' },
      $unset: {
        'synonymProcessingStatus.taskId': 1,
        'synonymProcessingStatus.startTime': 1
      }
    }
  );
};
```

#### 9.1.4 补漏机制

```typescript
// 在 processSynonymQueueTask 的主循环结束后

  task: DatasetSynonymQueueType,
  synonymConfig: SynonymConfigType,
  dataset: { vectorModel: string }

  const { teamId, datasetId, synonymFileIds } = task;  // ⚠️ 改为数组

  // 查找在处理期间新增的、未标准化的数据

    teamId,
    datasetId,
    skipSynonymStandardize: { $ne: true },
    // 没有 synonymMetadata 说明未被标准化

    $or: [
      { synonymMetadata: { $exists: false } },
      { synonymMetadata: null },
      { synonymMetadata: {} },
      { 'synonymMetadata.q': { $exists: false }, 'synonymMetadata.a': { $exists: false } }
    ],
    // 创建时间在任务开始之后
    createTime: { $gte: task.startTime }
  }).lean();

  if (newDataList.length > 0) {
    addLog.info(`[SynonymQueue] Mop-up scan found ${newDataList.length} new records`);

    for (const data of newDataList) {
      await standardizeDataItem(data, synonymConfig, synonymFileIds.map(String), dataset);  // ⚠️ 改为数组
    }
  }
```

### 9.3 数据一致性
- 使用 MongoDB Session 保证单条数据更新的原子性
- 断点续传机制保证任务中断后可恢复

### 9.4 性能考虑
- 分批处理（建议100条/批）
- 向量更新可考虑批量操作
- 大数据量时考虑并行处理多个 Worker
- 实时标准化是同步操作，需要控制延迟

### 9.5 回滚方案
- 保留 `synonymMetadata` 用于随时恢复
- 失败任务支持重试
- 可手动触发恢复任务

### 9.6 边界情况处理

| 边界情况 | 处理方式 |
|---------|---------|
| 队列任务执行中，用户新增数据 | 新数据实时标准化，不受队列影响 |
| 队列任务执行中，用户再次上传同义词 | 拒绝，提示"请等待当前任务完成" |
| 用户先标记 skipSynonymStandardize，后来又想标准化 | 提供"重新应用同义词"按钮 |
| 同义词更新后，部分数据已被新同义词处理 | update 模式先全量恢复再全量标准化 |
| **同义词队列处理中，training队列创建新数据** | 新数据跳过实时标准化，队列完成后补漏扫描 |
| **training队列处理中，用户上传同义词** | 同义词队列正常处理，补漏机制覆盖新数据 |

## 10. 总结

本方案通过新建独立的同义词处理队列（`dataset_synonym_queues`），实现了：

1. **异步处理**：不阻塞用户操作
2. **增量更新**：仅更新数据，不重新训练
3. **可恢复**：通过 `synonymMetadata` 支持原文恢复
4. **断点续传**：支持任务中断后继续
5. **进度可视**：实时查询处理进度
6. **用户自定义数据保护**：通过 `skipSynonymStandardize` 和开关控
### 场景覆盖总结

|-----|---------|---------|
| 首次上传同义词 | 异步队列 standardize | Worker |
| 更新同义词 | 异步队列 update（先恢复后标准化） | Worker |
| 删除同义词 | 异步队列 restore | Worker |

## 11. 遗漏场景补充

### 11.1 场景：直连数据库类型知识库不做标准化

**背景**：直连数据库类型的知识库（如 MySQL、PostgreSQL 等外部数据源）数据来自外部系统，不应该被同义词标准化修改。

**原因**：
1. 数据来源于外部数据库，FastGPT 无权修改原始数据
2. 外部数据可能随时同步更新，标准化后的数据会被覆盖
3. 直连数据库的数据结构和语义由外部系统定义，不应干预

**处理方案**：

| 操作 | 处理方式 |
|-----|---------|
| 上传同义词 | 直连数据库类型知识库**禁止上传同义词**，或上传后不对其数据生效 |
| 异步队列处理 | 查询时**排除直连数据库类型**的知识库数据 |
| 实时标准化 | 检测到直连数据库类型时**跳过标准化** |

**代码实现**：

```typescript
// 判断是否为直连数据库类型
const isExternalDatabase = (collectionType: DatasetCollectionTypeEnum) => {
  return [
    DatasetCollectionTypeEnum.externalFile,  // 外部文件
    // 添加其他直连数据库类型...
  ].includes(collectionType);
};

// 在实时标准化中跳过
export async function realtimeSynonymStandardize({ ... }) {
  // 检查是否为直连数据库类型
  const collection = await MongoDatasetCollection.findById(collectionId).lean();
  if (collection && isExternalDatabase(collection.type)) {
    // 直连数据库类型，不做标准化，直接返回原数据
    return { q, a, indexes, skipSynonymStandardize: true };
  }

  // ... 其他标准化逻辑
}

// 在异步队列处理中排除
const query = {
  teamId,
  datasetId,
  // 排除直连数据库类型的 collection
  collectionId: {
    $nin: await getExternalDatabaseCollectionIds(datasetId)
  },
  // ... 其他查询条件
};
```

**前端交互**：
- 直连数据库类型的知识库，同义词上传入口可以**隐藏**或**禁用**
- 或者上传时给出提示："直连数据库类型的数据不会应用同义词标准化"

### 11.2 场景：用户编辑已有分片的 q/a

**问题**：用户编辑一个分片时，如何处理？

**⚠️ 核心变更**：在新方案中，q/a 始终是原文，所以用户编辑时看到的就是原文。

```
用户编辑分片
    ↓
前端直接显示 data.q 和 data.a（原文）
    ↓
用户修改后保存
    ↓
根据开关决定是否对 indexes 标准化（**默认不标准化**）
```

**处理方案**：

| 情况 | 处理方式 |
|-----|---------|
| 编辑 q/a | 直接显示原文，保存原文，根据开关决定 indexes 是否标准化 |
| 新增 custom 索引 | 根据开关决定是否标准化（**默认不标准化**） |

**核心原则**：
1. ✅ **q/a 永远显示和保存原文**
2. ⚠️ **用户手动操作的 indexes，默认不标准化**
3. ⚠️ **用户需要主动开启开关才应用同义词**

**API 修改**：
```typescript
// projects/app/src/pages/api/core/dataset/data/update.ts
export type UpdateDataProps = {
  dataId: string;
  q?: string;   // ← 直接保存原文
  a?: string;   // ← 直接保存原文
  indexes?: DatasetDataIndexItemType[];
  applySynonym?: boolean;  // 新增：是否对 indexes 应用同义词（默认 false）
};

// 处理逻辑
async function updateData({ dataId, q, a, indexes, applySynonym = false }) {
  // 1. ✅ q/a 直接保存原文
  const updateFields: any = {};
  if (q !== undefined) updateFields.q = q;
  if (a !== undefined) updateFields.a = a;

  // 2. ✅ indexes 根据开关处理
  if (indexes) {
    if (applySynonym) {
      // 用户选择应用同义词
      const standardizedIndexes = await applyIndexSynonym(indexes, datasetId);
      updateFields.indexes = standardizedIndexes;
    } else {
      // 默认：不标准化，标记为用户自定义
      updateFields.indexes = indexes.map(idx => ({
        ...idx,
        skipSynonymStandardize: true  // ← 标记为用户自定义
      }));
    }
  }

  // 3. 更新数据库
  await MongoDatasetData.updateOne({ _id: dataId }, { $set: updateFields });

  // 4. ✅ 更新全文检索
  // 如果知识库有同义词配置，使用标准化后的 q/a 生成 token
  // 否则使用原文生成 token
  const hasSynonym = await checkDatasetHasSynonym(datasetId);
  if (hasSynonym) {
    const qStandardized = applySynonymTransform(q || data.q, ...);
    const aStandardized = applySynonymTransform(a || data.a, ...);
    await updateDataFullText({
      dataId,
      standardizedQ: qStandardized.transformedText,
      standardizedA: aStandardized.transformedText
    });
  } else {
    await updateDataFullText({
      dataId,
      standardizedQ: q || data.q,
      standardizedA: a || data.a
    });
  }
}
```

```typescript
  dataId,
  datasetId,
  collectionId,
  teamId,

  teamId: string;

  const existingText = await MongoDatasetDataText.findOne({ dataId }).lean();

  // 如果原来只有 q，就只用标准化后的 q
```

### 15.2 原文恢复算法

**已实现**：`restoreOriginalText` 函数（需要导出）

```typescript
/**
 * 根据转换记录恢复原始文本
 * 按transformedStartPos降序排列（从后向前恢复，避免位置偏移影响）
 *
 * @param transformedText 转换后的文本
 * @param transformations 转换记录数组
 * @returns 原始文本
 */
export function restoreOriginalText(
  transformedText: string,
  transformations: TransformationRecordType[]
): string {
  // 按transformedStartPos降序排列（从后向前恢复，避免位置偏移影响）
  const sortedTransforms = [...transformations].sort(
    (a, b) => b.transformedStartPos - a.transformedStartPos
  );

  let result = transformedText;

  // 从后向前依次恢复
  for (const transform of sortedTransforms) {
    const { transformedStartPos, transformedEndPos, originalTerm } = transform;

    // 替换：将标准词换回原始词
    const before = result.substring(0, transformedStartPos);
    const after = result.substring(transformedEndPos);
    result = before + originalTerm + after;
  }

  return result;
}
```

## 16. 已知限制与后续优化

### 16.1 当前方案限制

| 限制项 | 说明 | 影响 |
|-------|------|-----|
| 单知识库单任务 | 同一知识库同时只能有一个同义词处理任务 | 防止数据冲突，但大数据量时等待较长 |
| 实时标准化同步 | 新增数据时同步标准化 | 可能增加API延迟 |
| 位置敏感恢复 | 如果用户编辑了标准化后的文本，恢复可能不准确 | 需要在编辑前先恢复原文 |

### 16.2 后续优化方向

1. **批量向量化**：目前每条数据单独调用 embedding，可优化为批量处理
2. **并行 Worker**：支持多 Worker 并行处理不同知识库的任务
3. **增量标准化**：只处理与同义词相关的数据，而非全量
4. **编辑优化**：编辑时自动显示原文，无需用户手动恢复
